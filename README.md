# FAILSAFE

**Student failure-risk prediction system** — three calibrated XGBoost models (start-of-semester, mid-semester, end-of-semester), SHAP-based explanations, rule-based interventions, and a FastAPI + HTML web app.

Given a student's demographics, behaviour, and (optionally) their G1 / G2 grades, the system returns a calibrated risk probability, the top-5 features driving that score, and a concrete intervention plan. The UI compares risk across whatever models can run on the supplied inputs.

---

## Quick Start

```bash
# 1. Create venv + install deps
python3 -m venv .venv
source .venv/bin/activate
pip install -r backend/requirements.txt

# 2. Configure environment
cp backend/.env.example backend/.env
# (edit backend/.env if you want MySQL/Postgres or a custom SECRET_KEY)

# 3. Train the three models (writes models_bundle.pkl in repo root)
python backend/train_models.py

# 4. Run the backend
bash backend/start.sh
# → API at http://localhost:8000  (docs at /docs)

# 5. Open the frontend
open frontend/login.html
```

The default database is SQLite (`failsafe.db`), so no DB setup is needed. To use MySQL, set `DATABASE_URL` in a `.env` file (see [backend/database.py](backend/database.py)).

## The ML Pipeline

### Three stage-specific models

The single biggest design decision: faculty at the start of the semester have fundamentally different information from faculty who have already seen G1 and G2, so one model can't serve both. We train three:

| Stage  | When used               | Features                                                  | Held-out ROC-AUC |
|--------|-------------------------|-----------------------------------------------------------|------------------|
| `pre`  | Start of semester       | All categoricals + behaviour + background (no grades)     | **0.778**        |
| `mid`  | After G1 reported       | All `pre` features + G1                                   | **0.925**        |
| `full` | After both G1 + G2      | Everything, plus `avg_early_grade` and `grade_trend`      | **0.973**        |

The web UI auto-detects which models can run on the supplied inputs and plots the resulting risk scores side by side.

### Feature engineering choices

- **Pandas `category` dtype + XGBoost `enable_categorical=True`** — no arbitrary label-encoding. Trees learn optimal splits over actual category sets (`Mjob`, `Fjob`, `reason`, etc.) rather than over a meaningless integer order.
- **Six always-available engineered features** so the PRE model has signal to lean on once grades are stripped: `alcohol_total`, `parent_edu`, `high_absences`, `has_failures`, `social_load` (goout + freetime), `support_count` (count of school/family/paid support).
- **Two grade-derived features** added only when both grades are present: `avg_early_grade` and `grade_trend`.

### Probability calibration

Each stage trains **two** XGBoost objects:

1. A raw `XGBClassifier` — kept for SHAP (TreeExplainer needs the underlying trees).
2. A `CalibratedClassifierCV(method='isotonic', cv=5)` wrapper — used to serve probabilities.

Raw XGBoost probabilities tend to bunch near 0 / 1, making the "risk %" shown to faculty meaningless. Isotonic calibration on 5-fold CV maps the scores back to honest probabilities. Brier score (lower is better) confirms the improvement on every stage:

| Stage | Brier (raw) | Brier (calibrated) |
|-------|-------------|---------------------|
| pre   | 0.138       | **0.135**           |
| mid   | 0.102       | **0.090**           |
| full  | 0.065       | **0.058**           |

The PRE model also uses tighter regularisation (`max_depth=3`, `reg_lambda=2`) since with grades stripped out it's more prone to over-fitting noise.

## SHAP — Top Features per Stage

Mean |SHAP| value over a 150-row sample shows how each model leans on different signals:

| Rank | `pre`               | `mid`                | `full`                  |
|------|---------------------|----------------------|-------------------------|
| 1    | subject             | **G1** (dominates)   | **G2** (dominates)      |
| 2    | failures            | subject              | avg_early_grade         |
| 3    | school              | absences             | G1                      |
| 4    | absences            | failures             | subject                 |
| 5    | social_load         | social_load          | absences                |

The PRE model relies heavily on prior failures, subject, and school as proxies for academic readiness. `social_load` makes the top 5 — confirming the engineered feature is doing real work in the absence of grades. Behavioural features (`absences`, `social_load`, `traveltime`) appear across all three stages — they add complementary signal even when grades are present.

## API

All endpoints require a `Bearer <token>` header (obtain one via `POST /auth/login`).

| Method | Path                       | Purpose                                                  |
|--------|----------------------------|----------------------------------------------------------|
| POST   | `/auth/register`           | Create account                                           |
| POST   | `/auth/login`              | Get JWT (OAuth2 password form)                           |
| GET    | `/auth/me`                 | Current user                                             |
| POST   | `/predict`                 | Run risk prediction — see below                          |
| GET    | `/predictions`             | List the caller's predictions                            |
| GET    | `/dashboard/stats`         | Counts by risk level                                     |
| DELETE | `/predictions/{id}`        | Delete a prediction                                      |

### `POST /predict`

G1 and G2 are **optional**. The server auto-runs every model that fits the inputs, or you can pass an explicit `stages` list.

```json
{
  "student_name": "Rahul Verma",
  "school": "GP", "sex": "M", "age": 17, "address": "U",
  "famsize": "GT3", "Pstatus": "T",
  "Medu": 2, "Fedu": 1, "Mjob": "other", "Fjob": "other",
  "reason": "home", "guardian": "mother",
  "traveltime": 2, "studytime": 1, "failures": 0,
  "schoolsup": "no", "famsup": "yes", "paid": "no",
  "activities": "no", "nursery": "yes", "higher": "yes",
  "internet": "yes", "romantic": "no",
  "famrel": 3, "freetime": 4, "goout": 3,
  "Dalc": 1, "Walc": 2, "health": 4, "absences": 4,
  "subject": "math",
  "G1": 12
}
```

## Reproducing the Models

Two equivalent paths:

```bash
# (a) CLI — fast, no notebook needed
python backend/train_models.py

# (b) Notebook — full walkthrough with plots, EDA, calibration curves, SHAP
jupyter notebook Failsafe.ipynb
```

Both produce the same `models_bundle.pkl` — a dict with `pre` / `mid` / `full` keys, each holding `{model, calibrator, features, categorical, metrics}`.

## Limitations & Next Work

- The PRE model is **genuinely capped at ~0.78 AUC** by the dataset (no LMS interaction logs, no attendance time-series). Useful as a triage tool, not a precise classifier.
- `CORS allow_origins=["*"]` and a default dev `SECRET_KEY` — fine for development, must be tightened before any real deployment.
- No automated tests yet; no pagination on `/predictions`.
- Per-stage prediction history is not stored separately — only the latest-information stage is persisted per row. Adding a `stage` column to the `Prediction` table would enable per-student risk trajectories over the semester.

## Dataset

UCI Machine Learning Repository — *Student Performance Data Set* (Cortez & Silva, 2008). Two files (`student-mat.csv`, `student-por.csv`) merged into 1,044 rows with a `subject` tag. Target `at_risk = (G3 < 10)` on the 0–20 grading scale.
